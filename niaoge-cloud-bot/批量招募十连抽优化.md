# 批量招募十连抽优化

## 变更概述

将 `lib/batch/tasksItem.js` 中的 `recruit` 方法从**单抽循环**改为**十连分块**：

- 满 10 个的使用 `recruitNumber: 10` 一次招募 10 个
- 不足 10 个的余数使用 `recruitNumber: 1` 单抽
- 保持现有延迟、错误处理、`recruitType` 行为不变

## 涉及文件

- `lib/batch/tasksItem.js`

无需修改前端，`Accounts.vue` / `Batch.vue` / `Tasks.vue` 已经只传递 `maxCount`。

## 实现逻辑

```js
const total = Math.max(1, Number(maxCount) || 1);
let done = 0;
const tens = Math.floor(total / 10);
const remainder = total % 10;

// 整十部分十连抽
for (let i = 0; i < tens; i++) {
  await this.exec(accountId, "hero_recruit", { recruitType, recruitNumber: 10 }, `${typeName} ${done + 10}/${total}`, 8000);
  done += 10;
  await new Promise(r => setTimeout(r, 500));
}

// 余数部分单抽
for (let i = 0; i < remainder; i++) {
  await this.exec(accountId, "hero_recruit", { recruitType, recruitNumber: 1 }, `${typeName} ${done + 1}/${total}`, 8000);
  done += 1;
  await new Promise(r => setTimeout(r, 500));
}
```

## 行为示例

| 输入 | 十连请求数 | 单抽请求数 | 总请求数 | 耗时（仅延迟） |
|------|-----------|-----------|---------|--------------|
| 500  | 50        | 0         | 50      | ~25 秒       |
| 505  | 50        | 5         | 55      | ~27.5 秒     |
| 7    | 0         | 7         | 7       | ~3.5 秒      |

优化前 500 次单抽需要 500 次请求、约 250 秒延迟；优化后仅需 50 次请求、约 25 秒。

## 验证

前端构建已通过：

```bash
cd frontend && npm run build
```

运行时验证建议：

1. 输入 `500`，日志应显示 `付费招募 10/500` 到 `付费招募 500/500`。
2. 输入 `505`，日志应先显示 `付费招募 10/505` 到 `付费招募 500/505`，再显示 `付费招募 501/505` 到 `付费招募 505/505`。
3. 输入 `7`，日志应显示 `付费招募 1/7` 到 `付费招募 7/7`。
4. 招募令不足时，应在失败处停止并输出错误日志。

## 风险

- 假设 `recruitNumber: 10` 请求是原子的：要么成功 10 次，要么失败。这与单抽逻辑一致。
- `maxCount` 非数字或小于 1 时回退到 1，保持与现有行为一致。
- 超时保持 8000ms，若服务端十连返回慢可适当上调。
