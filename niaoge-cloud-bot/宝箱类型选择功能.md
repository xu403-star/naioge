# 宝箱类型选择功能

## 变更概述

在批量开箱操作中增加宝箱类型选择，支持 **木质 / 青铜 / 黄金 / 铂金** 四种宝箱。

## 涉及文件

- `frontend/src/views/Accounts.vue`
- `frontend/src/views/Batch.vue`
- `frontend/src/views/Tasks.vue`

> 后端接口 `lib/batchEngine.js` 已原生支持 `body.boxId`，无需修改。

## 宝箱类型映射

| value | label |
|-------|-------|
| 2001  | 木质  |
| 2002  | 青铜  |
| 2003  | 黄金  |
| 2004  | 铂金  |

对应后端常量：`lib/batch/tasksItem.js` 中的 `BOX_NAMES`。

## 前端实现要点

### 1. 扩展 `countConfig`

为 `chest` / `itemOpenBox` 操作增加 `options` 配置：

```js
const boxTypes = [
  { value: 2001, label: '木质' },
  { value: 2002, label: '青铜' },
  { value: 2003, label: '黄金' },
  { value: 2004, label: '铂金' },
]

const countConfig = {
  chest: {
    label: '开箱数量',
    field: 'maxCount',
    default: 100,
    options: boxTypes,
    optionsField: 'boxId',
    optionsLabel: '宝箱类型',
    optionsDefault: 2001,
  },
  fish: { label: '钓鱼次数', field: 'maxCount', default: 100 },
  recruit: { label: '招募次数', field: 'maxCount', default: 10 },
}
```

### 2. 弹窗增加下拉选择

在数量输入弹窗中，当 `countConfig[operation]?.options` 存在时渲染 `<select>`：

```vue
<div v-if="countConfig[countOperation]?.options" class="input-group">
  <label>{{ countConfig[countOperation]?.optionsLabel }}</label>
  <select v-model.number="selectedBoxType">
    <option v-for="opt in countConfig[countOperation]?.options" :key="opt.value" :value="opt.value">
      {{ opt.label }}
    </option>
  </select>
</div>
```

### 3. 提交时附带 `boxId`

```js
function confirmCountInput() {
  const cfg = countConfig[countOperation.value]
  const body = { [cfg.field]: Math.max(1, countValue.value || cfg.default) }
  if (cfg.options && cfg.optionsField) {
    body[cfg.optionsField] = selectedBoxType.value || cfg.optionsDefault
  }
  pendingCountResolve.value(body)
}
```

最终请求体示例：

```json
{
  "accountIds": ["xxx"],
  "maxCount": 100,
  "boxId": 2003
}
```

```
POST /api/batch/run-all/:operation
```

`lib/batchEngine.js` 中 `chest` / `itemOpenBox` 分支：

# WebSocket 会话恢复机制说明

## 控制开关

文件：`lib/connectionPool.js:29`

```js
this.enableSessionRestore = false;
```

## 两种重连模式对比

游戏服务器使用两套身份机制：

- `roleToken` / `roleId`：证明账号身份。
- `sessId` / `connId`：证明本次会话身份。

### 模式一：默认关闭（`false`）

每次断线都当作「新登录」：

1. 用数据库里最新 Token 直接连接。
2. 只有服务端明确返回 `token expired` 时才调用 `refreshToken` 从 BIN/URL 拿新 Token。
3. 普通网络抖动（`code=1006`）不重刷 Token，直接用旧 Token 重试。

**例子：**

账号 A 跑每日任务时网络抖动断开，系统用旧 Token 重新登录，服务端创建新会话，任务继续跑。如果旧 Token 在服务端已被标记失效，会连续失败，直到触发 `token expired` 才走完整刷新。

### 模式二：开启（`true`）

断线后优先「会话恢复」：

1. 保持上次 `sessId` 不变。
2. `isRestore = 1`，`connId = Date.now()` 换成新的。
3. URL 附加 `&ack=上次收到的 seq`。

这告诉服务器：我不是新登录，我是刚才那个连接断了，请续接旧会话并补发遗漏消息。

**例子：**

账号 A 跑任务时网络抖动断开，系统用 `sessId=1234567890`、`connId=新的`、`ack=上次 seq` 重新连接。服务器认出同一会话，直接恢复，不需要重新初始化。

## 对比总结

| 场景 | `false` | `true` |
|---|---|---|
| 网络抖动断线 | 用旧 Token 重新登录 | 用恢复 Token 续接旧会话 |
| Token 被服务端明确过期 | 都会触发 `refreshToken` 完整刷新 | 一样 |
| 服务器负载 | 每次重连都是完整登录 | 恢复会话更轻量 |
| 兼容性 | 最稳妥，所有环境都能连 | 部分账号/网络环境可能不稳定 |
| 任务连续性 | 可能丢失断线期间响应 | 理论上可补发 ack 之前的消息 |

## 为什么默认关闭？

代码注释说明：会话恢复在部分账号/网络环境下不稳定，默认关闭以优先保证连接成功率。

开启后可能遇到的问题：

- 服务器不认这个 `sessId`，恢复失败，再回退到普通连接。
- 某些代理/负载均衡下 `ack` 机制表现异常。

## 什么时候该开启？

满足以下条件时可以尝试：

- BIN 长期有效、不会过期。
- 网络偶发抖动较多。
- 跑长时间任务（如批量开箱 1000 个）时不想频繁重新登录。

建议改成 `true` 测试几天。稳定则保持开启；若重连失败变多，再改回 `false`。

## 如何开启

修改 `lib/connectionPool.js:29`：

```js
this.enableSessionRestore = true;
```
